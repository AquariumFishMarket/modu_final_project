<h1 align="center">물고기마켓 (Aquarium Fish Market)</h1>

<p align="center">
  관상어 분양 및 일상 공유에 특화된 모바일 중심의 SNS형 커뮤니티/거래 플랫폼
</p>

<br>

물고기마켓은 **관상어 분양, 어항 용품 공유, 반려 물고기 관련 정보 교류**에 최적화된 모바일 기반 커뮤니티/거래 플랫폼입니다.  
React/Vite/TypeScript 기반으로 개발되었으며, **피드·좋아요·팔로우·댓글·프로필·검색** 등 SNS 핵심 기능은 물론  
**실시간 채팅(UI), 플로팅 챗봇, 에러 페이지, 스플래시 화면**까지 갖춘 완성형 모바일 SNS 구조를 목표로 구축했습니다.

현재는 위니브에서 제공한 **감귤마켓 API**로 연동되어 있으며, 추후 자체 백엔드 서버를 구축하여 실서비스 수준으로 확장할 예정입니다.

## 프로젝트 개요

### 프로젝트 목적

- 관상어/어항등 분양/용품에 특화된 **커뮤니티 + 거래 플랫폼** 구축
- 모바일 우선(UI/레이아웃)으로, 가볍고 직관적인 사용 경험 제공
- 기존 SNS(인스타, 당근, 오늘의집)의 **피드/좋아요/팔로우/댓글 패턴**을 적용
- **챗봇, 채팅, 팔로워/팔로잉 관리** 등 고도화 기능까지 고려한 프론트 설계
- 현재는 제공 API 연동, 이후 **자체 백엔드 구현**으로 기능 확장 예정

## 기술 스택

### Frontend

- **React 18**  
  최신 렌더링 아키텍처 및 훅 개선 사항을 활용하여 전체 UI 성능과 개발 생산성을 향상하기 위해 선택했습니다.

- **Vite**  
  매우 빠른 개발 서버와 최적화된 빌드를 제공하여 개발 효율을 극대화하기 위해 사용했습니다.

- **TypeScript**  
  안전한 타입 시스템을 기반으로 런타임 오류를 줄이고 유지보수성을 높이고자 도입했습니다.

- **Zustand**  
  가볍고 직관적인 API를 가진 전역 상태 관리 라이브러리로, Redux 대비 복잡성이 적고  
  피드/좋아요/프로필/팔로우 등 다양한 전역 상태를 간결하게 관리할 수 있다는 장점 때문에 선택했습니다.

- **Styled-Components**  
  컴포넌트 단위로 스타일을 관리할 수 있어 UI 재사용성과 유지보수성을 높여줍니다.  
  모바일 중심 UI 설계에도 최적화되어 있어 선택했습니다.

- **React Router v6**  
  Nested Routes와 Guard 기능 관리에 용이해, 페이지 구조 관리 및 접근 제어 구현에 적합하여 사용했습니다.

- **Framer Motion**  
  가벼운 전환 효과 및 인터랙션 개선을 위해 적용했습니다. 주요 UI 요소의 자연스러운 전환에 도움을 줍니다.

- **Axios**  
  서비스 레이어 기반의 API 모듈화를 위해 사용했으며, 인증 토큰 주입 및 에러 처리 구조를 명확히 하기 용이합니다.

- **ESLint / Prettier**  
  일관된 코드 스타일을 유지하고 팀 협업 시 코드 품질을 보장하기 위해 설정했습니다.

## 화면 구성

####  1) 스플래시 & 인증 라우팅

  - 앱 진입 시 스플래시 노출

  - tokenManager로 저장된 토큰 확인

  - MemberRoute / GuestRoute 기반 접근 제어

  - 로그인 유지 → 바로 피드 페이지로 이동

  → UX 좋은 실제 앱 구조

#### 2) 피드 페이지
**기능**
- Pull To Refresh
- 피드 랜덤 배치
- 무한 스크롤
- 로딩/마지막 페이지 안내
- Top 버튼

**기술적 핵심**
- IntersectionObserver 기반 무한 스크롤
- isFetching / isRefreshing / hasMore 로 요청 제어
- Race Condition 방지

#### 3) 게시글 상세 페이지

  - 댓글 작성 즉시 증가

  - 좋아요 낙관적 업데이트 (성공 실패에 따라 롤백)

  - 댓글/좋아요 메타 정보 중앙화(fetchPostMeta)

  → 실제 SNS와 동일한 상호작용 구조

#### 4) 검색 기능
**기능**
- 2글자 이상 입력 시만 검색 수행
- Debounce 300ms
- 검색어 하이라이트
- 결과 없음 UI

**기술적 핵심**
- AbortController로 이전 요청 중단
- requestId로 최신 요청만 반영
- race-condition 완전 해결

#### 5) 프로필 페이지
- 닉네임·계정명·소개글
- 팔로워/팔로잉 목록
- 판매 상품 및 게시글
- 갤러리형 게시글
- 외부 링크 이동
- 데이트 포매터 (방금 전, 1시간 전)

**기술 포인트**
- followStore로 팔로우 상태 전역관리
- myInfo 실시간 갱신 구조

#### 6) 게시글 작성 페이지
- 최대 10장 이미지 업로드
- 이미지 삭제 가능
- 텍스트만 작성 가능
- 이미지 업로드 실패 처리 강화

#### 7) 채팅 UI
- Mock 기반 UI 구성
- 시간차 메시지 렌더링
- Socket.io 기반 실시간 기능을 고려한 구조

#### 8) 404 페이지
- 귀여운 물고기 애니메이션 추가
- Soft UX 제공

#### 9) 챗봇
- Intent + Synonym 기반 Rule Engine
- FAQ 매칭
- 플로팅 챗봇 (앱 어디서든 접근 가능)
- KT 챗봇 구조 참고

## 주요 구현 포인트

물고기마켓은 단순 기능 구현이 아니라 **SNS 플랫폼 구조를 실제로 어떻게 설계하고 운영할 수 있는가**에 초점을 두어 개발했습니다.  
아래는 그중 핵심이 되는 기술적 구현 포인트들입니다.

---

### 1) 상태 관리 (Zustand + Context) 구조화

물고기마켓의 전역 상태는 Redux 대비 가벼운 **Zustand**로 관리하였습니다.  
각 도메인별 스토어를 분리하여 유지보수성과 확장성을 높였습니다.

#### ● useFeedStore — 피드/좋아요/무한 스크롤 핵심 로직

- feedList, skip, hasMore, isFetching 등 **무한스크롤 전용 상태**를 가진 독립 스토어
- 좋아요 토글 시 서버 응답을 기다리지 않고 **Optimistic Update(낙관적 업데이트)** 적용

## 협업 구조 변화
처음에는 **페이지 단위(feature-based)**로 작업했으나,
API/상태 관리가 늘어나면서 중복 코드 증가 데이터 일관성 깨짐 책임 모호 문제가 발생했습니다.
그래서 중반부터 **도메인 기반(domain-based)**으로 전환:

**예:**

- 검색/좋아요/팔로우/챗봇 → 1명

- 인증/프로필/공통 컴포넌트 → 1명

- 채팅/전역 상태/라우팅 → 1명

**효과:**

- 한 기능의 API → Store → UI를 한 개발자가 책임

- 충돌 감소

- 속도 증가

- 실서비스처럼 효율적인 구조

## 트러블슈팅 (Trouble Shooting)

프로젝트 개발 과정에서 다양한 기술적 문제를 경험했고,  
각 문제를 단순히 해결하는 데 그치지 않고 **근본 원인 분석 → 구조적 해결 → 원칙 도출**까지 진행했습니다.  
아래는 실제로 해결한 핵심 이슈들입니다.

---

### 1) API 응답 필드 불일치 (hearted vs isLiked)

**문제**  
  빠르게 타이핑하면 검색 결과가 뒤섞임
"결과 없음"과 "API 오류"가 랜덤하게 섞여 UI 깜빡임
1글자 입력에도 불필요한 API 요청이 계속 호출
실제 사용자에게 "검색이 불안정하다"는 피드백 발생
- 좋아요 토글 시 UI가 즉시 반응하지 않음
- 화면 새로고침 시 좋아요 상태가 뒤바뀌어 보임  
  과 같은 문제가 발생했습니다.

**원인**  
1. 레이스 컨디션 발생
  * 요청이 여러 개 동시에 전송
  * 느린 응답이 최신 검색 결과를 덮어버림
2. AbortController 타이밍 문제
  * abort 호출 전에 새 요청이 실행됨
    → 중단된 요청이 완전히 사라지지 않고 응답이 들어옴
3. 디바운스 이전에 불필요한 호출
  * 1글자/공백 입력도 API로 전달
    → 서버 404 + UI 깜빡임 빈번
4. 404(검색 결과 없음)를 오류로 처리함
  * “에러”로 분류 → UI가 항상 에러 상태로 변경됨

**해결**  
1) AbortController로 이전 요청 즉시 취소
```typescript
if (abortControllerRef.current) {
  abortControllerRef.current.abort();
}
```
→ 중복 요청 제거
→ 빠른 입력에도 불필요한 응답 방지

2) 요청 ID(searchIdRef)로 최신 요청만 반영
```typescript
if (requestId !== searchIdRef.current) return null;
```
→ 느린 응답이 최신 검색 결과를 덮는 문제 해결
→ 레이스 컨디션 근본 차단

3) 2글자 미만 입력은 API 호출 차단 + 디바운스 300ms
```typescript
if (!trimmed || trimmed.length < 2) return;
```
→ 1글자 입력 시 API 호출 완전 차단
→ API 호출 횟수 대폭 감소
→ UX 부드러워짐

4) “결과 없음(404)”은 정상 흐름으로 처리
```typescript
if (response.status === 404) return [];
```
→ UI 에러 상태 전환 제거
→ 자연스럽게 “결과 없음” 화면 렌더링


#### 배운 점 / 적용 원칙
백엔드(hearted)와 프론트(isLiked)의 스키마 불일치가 UI 버그의 핵심 원인
그래서 API 응답을 UI 전용 스키마로 변환하는 매핑 레이어(Data Adapter)를 추가
이후 UI, Store, LocalStorage는 isLiked 하나만 사용하도록 통일

이 방식으로 좋아요 상태가 모든 화면에서 일관되게 유지되고 유지보수가 훨씬 쉬워졌다.

---

### 2) 검색 기능 오류

**문제**  
  빠른 타이핑 시 요청이 뒤섞임(레이스 컨디션)
  1글자/공백 입력도 API 호출
  404(결과 없음)도 에러로 처리 → UI 깜빡임
  이전 요청 abort 처리 지연 → 결과가 뒤엉킴

  검색 자체가 불안정한 상태였음.

**원인**  
  AbortController가 동시 요청을 완전히 중단시키지 못함
  requestId 검사 부재
  디바운스 타이머가 제대로 초기화되지 않음
  서버 응답 메시지 불일치로 UI가 매번 다른 상태로 진입

**해결**  
  AbortController로 이전 요청 즉시 취소
  requestId(currentRequestId) 비교로 최신 요청만 반영
  2글자 미만 입력 시 요청 차단
  404는 에러가 아니라 “검색 결과 없음”으로 정상 처리

```ts
handleInputChange(value) {
  const trimmed = value.trim();
  if (trimmed.length < 2) return resetResults();

  // 디바운스 적용
  clearTimeout(timer);
  timer = setTimeout(() => performSearch(trimmed), 300);
}

async function performSearch(keyword) {
  // 1. 요청 번호 증가
  searchIdRef.current++;
  const requestId = searchIdRef.current;

  // 2. 이전 요청 abort
  abortControllerRef.current?.abort();
  const controller = new AbortController();
  abortControllerRef.current = controller;

  const res = await fetchUsers(keyword, controller);

  // 3. 최신 요청만 반영
  if (requestId !== searchIdRef.current) return;

  setSearchResults(res || []);
}
```

#### 배운 점 / 적용 원칙

  검색처럼 비동기 요청이 많은 UI는
  abort + requestId + 최소 글자 수 조건만 잘 적용해도
  대부분의 race 문제를 해결할 수 있음.

### 3) 팔로워 / 팔로잉 수 UI 불일치 문제

**문제**
팔로우/언팔로우 버튼을 눌러도:

팔로워 수가 즉시 반영되지 않음

새로고침해야 정확한 수가 보여짐

프로필 페이지와 팔로워 리스트 간 데이터가 다르게 보임

**원인**
팔로우 관련 상태가 여러 컴포넌트에서 각각 관리되어
전역 동기화가 되지 않음.

**해결**

팔로우 상태를 **followStore(Zustand)**로 전역 관리

팔로우/언팔로우 성공 시

`followStore` 업데이트

`myInfo` 재요청

프로필 페이지·리스트 페이지가 즉시 새 값 반영

#### 배운 점 / 적용 원칙

“서로 다른 페이지에서 공유되는 상태”는 반드시 중앙 Store에서 관리해야 일관성 유지

API 성공 시 UI 업데이트를 즉시 반영해야 최신 SNS 사용자 경험에 부합함

## 팀원 기여도

프로젝트 초반에는 UI 개발이 중심이었기 때문에  
**페이지 단위로 컴포넌트·레이아웃을 나누는 방식(feature-based)**으로 작업을 진행했습니다.

하지만 API 연동과 상태 관리가 본격적으로 필요해지면서  
페이지 단위 분업만으로는 데이터 흐름과 기능 연결성이 떨어졌고, 유지보수성이 낮아지는 문제가 생겼습니다.

이에 따라 개발 중반부터는  
**피드/좋아요, 프로필·팔로우, 검색, 채팅·챗봇 등 기능별 도메인 기반 역할 분담(domain-based)**으로 전환했습니다.

이러한 전환을 통해 다음과 같은 개선이 이루어졌습니다:

- API → 전역 상태 → UI까지 **전체 흐름을 한 개발자가 책임**
- 데이터 일관성 향상
- 충돌/중복 작업 감소
- 유지보수성과 개발 속도 증가
